# Задание 2.1: API-тесты

В репозитории реализовано решение варианта `2.1: API-тесты` для сервиса объявлений `https://qa-internship.avito.com`.

Что внутри:

- автоматизированные API-тесты на `pytest`
- E2E-сценарии на создание, получение, список продавца и статистику
- дополнительное покрытие `DELETE /api/2/item/{id}` из Postman-коллекции, включая негативные сценарии
- Allure-аннотации: `title`, `description`, `epic`, `feature`, `story`, шаги
- конфигурация линтера и форматтера через `ruff`
- документация по тест-кейсам и шаблон баг-репортов

Быстрая навигация:

- [Описание тест-кейсов](./docs/TESTCASES.md)
- [Баг-репорты](./docs/BUGS.md)

## Состав решения

- `tests/test_ads_api.py` — основной набор автоматизированных тестов
- `tests/conftest.py` — фикстуры, генерация данных и CLI-параметры pytest
- `src/api_client.py` — HTTP-клиент с обработкой SSL и таймаутов

- `requirements.txt` — Python-зависимости
- `pytest.ini` — конфигурация pytest и маркировка тестов
- `pyproject.toml` — конфигурация `ruff`

- Python `3.7+`

## Установка зависимостей

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

## Запуск тестов

Полный прогон:

```bash
pytest
```

Если стенд отвечает медленно, имеет смысл сразу запускать так:

```bash
pytest --request-timeout 30
```

Если стенд отдает проблемный сертификат, текущая конфигурация уже запускает запросы без строгой SSL-проверки. Если нужно включить нормальную проверку сертификата:

```bash
pytest --verify-ssl
```

Запуск только обязательных позитивных smoke/E2E сценариев:

```bash
pytest -m "smoke or e2e"
```

Запуск негативных проверок:

```bash
pytest -m negative
```

Запуск с кастомным base URL:

```bash
pytest --base-url https://qa-internship.avito.com
```

Порог для легкой нефункциональной проверки времени ответа можно менять:

```bash
pytest --max-response-time 5.0
```

Технические параметры сетевого клиента:

```bash
pytest --request-timeout 20
```

## Allure

Запуск с сохранением результатов:

```bash
pytest --alluredir=allure-results
```

Генерация HTML-отчета:

```bash
allure generate allure-results --clean -o allure-report
```

Открытие локально:

```bash
allure open allure-report
```

Примечание: для команд `allure` должен быть установлен Allure CLI отдельно от Python-зависимостей.

## Линтер и форматирование

Конфигурация хранится в `pyproject.toml`.

Проверка линтером:

```bash
ruff check .
```

Автоисправление части замечаний:

```bash
ruff check . --fix
```

## Что именно проверяют тесты

- корректное создание объявления
- получение объявления по `id`
- получение списка объявлений продавца
- получение статистики объявления
- создание двух объявлений с одинаковым payload и разными `id`
- валидацию обязательных полей
- валидацию типов данных
- обработку несуществующих `id`
- обработку невалидного `sellerID`
- базовую проверку времени ответа
- удаление объявления через `api/2`
- негативные сценарии удаления: невалидный и несуществующий `id`

## Важные договоренности решения

- Для POST используется поле `sellerID`, потому что именно оно указано в Postman-коллекции.
- В ответах валидируется поле `sellerId`, потому что именно так оно возвращается в примерах коллекции.
- Для изоляции тестов используется уникальный `sellerID` в диапазоне `111111–999999`.
- Каждый тест независим: сам подготавливает данные и не зависит от порядка запуска.
- По умолчанию SSL-проверка сертификата отключена, чтобы тесты стабильно работали на учебном стенде.
- `DELETE /api/2/item/{id}` оставлен в решении как дополнительное покрытие по коллекции и как демонстрация E2E-подхода.

## Если сервис ведет себя иначе

Если в окружении проверки обнаружатся расхождения между фактическим поведением API и контрактом из Postman-коллекции, их нужно зафиксировать в `docs/BUGS.md`.
