# TESTCASES

## Подход

Цель набора тестов — не перебрать все возможные комбинации входных данных, а с минимально достаточным количеством проверок подтвердить, что API:

- корректно работает по основному пользовательскому сценарию
- валидирует обязательные поля и базовые типы данных
- не путает данные разных сущностей
- корректно обрабатывает ошибки и отсутствующие записи
- выдерживает базовые нефункциональные ожидания

Для покрытия использованы техники тест-дизайна:

- классы эквивалентности
- анализ граничных значений
- негативное тестирование
- переходы состояний для E2E-сценариев
- предположение об ошибках

Набор покрывает позитивные, негативные, корнер- и нефункциональные проверки.

## Почему выбран именно такой набор

Для данного задания у сервиса маленький и хорошо очерченный контракт: четыре основные ручки из условия и дополнительная ручка удаления из Postman-коллекции. Для такого API достаточное покрытие должно отвечать на четыре главных вопроса:

1. Можно ли успешно создать и затем получить сущность всеми предусмотренными способами.
2. Отфильтровывает ли сервис данные корректно: по `id`, по `sellerID`, по статистике конкретного объявления.
3. Защищен ли API от очевидно невалидных запросов.
4. Не ломается ли поведение на типичных рисковых сценариях: повторная отправка одинакового payload, запрос несуществующих данных, базовая проверка времени ответа.

## Почему этого достаточно для задания

Этот набор можно считать достаточным для тестового задания по следующим причинам:

- покрыты все обязательные ручки из условия
- для каждой ручки есть как минимум один успешный сценарий
- для ручек с входными параметрами проверены негативные классы эквивалентности
- добавлены E2E-проверки, подтверждающие связность сценариев между ручками
- отдельно проверено поведение на несуществующих сущностях
- отдельно проверен corner-case на неидемпотентность создания
- добавлена базовая нефункциональная проверка времени ответа
- зафиксировано расхождение между контрактом Postman и фактическим поведением сервиса в [BUGS.md](BUGS.md)

При этом часть кейсов осознанно не автоматизирована, потому что бизнес-правила для них не заданы в спецификации. Это не пробел в покрытии, а осознанное решение: такие проверки лучше оформлять как exploratory / исследовательские и при необходимости выносить в баг-репорты.

## Область тестирования

Основные ручки из условия:

- `POST /api/1/item`
- `GET /api/1/item/{id}`
- `GET /api/1/{sellerID}/item`
- `GET /api/1/statistic/{id}`

Дополнительно включено в покрытие по Postman-коллекции:

- `DELETE /api/2/item/{id}`

## Обоснование по группам тестов

### Позитивные сценарии

Позитивные сценарии `TC-01` — `TC-04` проверяют полный жизненный цикл сущности:

- создание записи
- получение конкретной записи по `id`
- получение набора записей продавца
- получение статистики конкретного объявления

Этого достаточно, чтобы подтвердить работоспособность всех обязательных ручек и их согласованность между собой.

### Corner-case сценарии

`TC-05` нужен, потому что в условии прямо упомянута идемпотентность как пример важного corner-case. Здесь мы проверяем, что повторная отправка одинакового тела не затирает предыдущее объявление и создает новую сущность с новым `id`.

### Негативные сценарии для POST

`TC-06` — `TC-13` покрывают два самых важных класса ошибок входных данных:

- отсутствие обязательного поля
- неверный тип данных

Этого достаточно как базового уровня валидации, потому что именно эти дефекты наиболее вероятны и именно они чаще всего приводят к некорректной сериализации, ошибкам бизнес-логики или невалидным данным в хранилище.

### Негативные сценарии для GET

`TC-14` — `TC-16` проверяют, что сервис различает:

- несуществующую, но формально корректную сущность
- некорректный формат path-параметра

Это важное разделение, потому что оно показывает корректность серверной валидации и семантики кодов ответа.

### Нефункциональная проверка

`TC-17` не претендует на полноценное нагрузочное тестирование, но достаточен как smoke-проверка производительности: если даже простая операция чтения не укладывается в адекватный порог, это уже сигнал о деградации стенда или проблеме производительности.

### Проверки удаления

Ручка удаления не перечислена в ядре из 4 ручек в формулировке задания, но присутствует в выданной Postman-коллекции, поэтому она оставлена в решении как дополнительное покрытие и как демонстрация E2E-подхода.

`TC-18` — `TC-20` проверяют:

- успешное удаление существующей сущности
- валидацию формата идентификатора
- корректную обработку отсутствующей сущности

Это делает покрытие ручки удаления полноценным и согласованным с примерами ответов в коллекции.

## Оценка полноты и корректности набора

Текущий набор тестов является корректным и достаточным для сдачи задания, потому что:

- не содержит явных дублирующих проверок без добавочной ценности
- покрывает основной happy path и критичные негативные сценарии
- соответствует имеющемуся контракту и отдельно фиксирует найденные расхождения
- остается воспроизводимым и независимым
- дополнительно покрывает `DELETE /api/2/item/{id}` не только позитивно, но и негативно

Что сознательно не включено в обязательную автоматизацию:

- глубокие проверки бизнес-ограничений на диапазоны значений
- нагрузочное и конкурентное тестирование
- массовые комбинации специальных символов и длинных строк

Эти сценарии полезны, но для тестового задания они уже лежат в зоне расширенного исследовательского покрытия, а не минимально необходимого regression/smoke-набора.

## Дополнительные ручные проверки, которые стоит сделать при живом исследовании сервиса

Ниже кейсы включены в тестовую модель, но не автоматизированы, чтобы не делать решение излишне хрупким без предварительного исследования среды:

- пустая строка в `name`
- `price = 0`
- отрицательные значения в `price` и статистике
- очень длинный `name`
- специальные символы и Unicode в `name`
- повторное удаление одного и того же объявления
- конкурентное создание нескольких объявлений одним продавцом
- нагрузочный прогон серии запросов

## Почему часть кейсов оставлена ручными

Некоторые проверки зависят от бизнес-правил, которые не зафиксированы в Postman-коллекции. Например, коллекция не уточняет:

- допустимы ли нулевые или отрицательные значения числовых полей
- есть ли ограничение на длину `name`
- должен ли пустой список по продавцу возвращать `200` или `404`

Такие кейсы лучше подтверждать исследовательским тестированием и при расхождении заводить баг-репорт.
